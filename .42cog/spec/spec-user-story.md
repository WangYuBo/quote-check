# 用户故事规约（User Stories）

## 0. 本规约读法

- **上游**：`spec-product-requirements.md` 的 6 个 MAS（Minimum Affordance Story）定义"系统应提供哪些可感知的行动机会"
- **本规约**：把 MAS 拆为**用户视角的最小价值单元**，每条带"三极叙事分类 + 验收准则 + 依赖 + MVP 复用度"
- **下游**：架构规约按 MS 设计组件；开发按 MS 排期；QA 按 MS 验收准则写测试

**术语速查**：
- **CS**（Complex Story）：复合故事，若干 MS 组成的完整用户旅程
- **MS**（Minimal Story）：最小故事，1-3 人日可完成的独立价值单元
- **L / D / G**：Light（逆境转顺境）/ Dark（顺境遇阻→恢复）/ Grey（循环日常操作）
- **MVP 复用度**：`已有`（原封移植）/ `改造`（逻辑保留但需重构）/ `全新`（MVP 未实现）

## 1. 用户类型（对齐 cog.md + meta.md）

### 1.1 两类核心编辑用户（共享 affordance，差异化交互皮肤）

| 用户类型 | 来源 | 典型场景 | 付费意愿 | 合规敏感度 |
|---------|------|---------|---------|-----------|
| **B 端质检编辑** | 高校/文史类出版社 | 一本书稿的质检流水作业、多席位协同 | 高（席位制） | 极高（数据流向协议、TTL 销毁、留痕审计） |
| **C 端研究生/自出版学者** | 人文社科研究生 + 独立作者 | 单本学位论文/个人书稿、偶发高密度使用 | 中（按次付费） | 中（隐私敏感但无合规审计需求） |

**设计原则**：同一套 affordance 覆盖两类用户，通过**交互皮肤**差异化（B 端多一层席位登录 / 协议弹窗 / 审计日志入口；C 端轻量、单机心智）。不做双产品。

### 1.2 非主要 agent（v1.0 不针对，留接口）

- **下游 AI agent**（用户自己的 Claude Code / Cursor / MCP 客户端调用本系统）：v1.1+ 开放（MAS-候选-7）
- **系统管理员**（SaaS 化后）：v2.0 再考虑席位管理、用量对账

## 2. 三极叙事分布策略

### 2.1 本项目偏离"标准 MVP 70/20/10"的理由

SKILL.md 建议 MVP 阶段 Light 70% / Dark 20% / Grey 10%。**本项目不照搬**——quote-check 的用户信任建立**依赖 Dark 场景处理得好**：

- 模型审核拒绝（notes #1）——国产 AI 的真实事故源
- API 限流 / 超时 / 网络错误——asyncio + 硅基流动的常态
- 引文在参考中找不到 / 不符合——MAS-2 的三态核心之一
- 费用超预估自动暂停（real.md #6）——商业安全底线
- 任务崩溃 / 断点续跑（notes #4）——工程信任底线

**故本项目取 Light 45% / Dark 35% / Grey 20%**——Dark 故事是**信任基础设施**而非点缀。

### 2.2 目标分布

| 类型 | 占比 | 绝对数（约 22 MS） | 角色 |
|------|------|-------------------|------|
| Light | 45% | ~10 | 价值递交、首次成功 |
| Dark | 35% | ~8 | 异常处理、信任建立 |
| Grey | 20% | ~4 | 重复使用、效率优化 |

### 3.3 用户旅程（CS 串联）

```
新用户 → 注册/登录（CS-00）

  → 看到费用预估 → 确认并发起（CS-01 + CS-02 的成本保护）
  → SSE 观察进度 → 遇到审核拒绝 / API 错误 → 系统显式处理（CS-02）
  → 看到三维度独立报告 + 参考匹配三态标记（CS-01 后段）
  → 人工终审每条（CS-05）
  → 导出 Word/CSV（CS-05）
  → 任务完成，N 天后自动销毁（CS-04）
  → 日后回看报告，版本戳依然完整（CS-03）
```

## 4. 详细故事清单

### CS-00：账户与角色基础

**描述**：编辑首次进入系统，完成身份识别与角色区分。B/C 端共享底层账户体系，差异在角色字段与协议弹窗。

**Core Value**：让编辑进入"可发起任务"的状态；角色字段决定后续 affordance 差异化。

---

#### MS-L-01：编辑注册账户（区分 B/C 端角色）

**CS**：CS-00 | **Type**：Light | **Priority**：P1 | **Story Points**：3 | **MVP 复用度**：全新

**User Story**:
作为一个 **未使用过本系统的编辑**（B 端质检 / C 端研究生/自出版学者），
我想 **用邮箱注册并声明自己的角色类型**，
以便 **获得适配我的交互皮肤与数据合规待遇**。

**Light Story 评估**:
- **起始状态**：没有账户，无法上传书稿
- **转折点**：注册表单区分"机构质检" / "个人研究/自出版"两类
- **终止状态**：账户创建成功，进入工作台，角色已落为 cog.md `用户.角色` 字段
- **情感弧**：陌生 → 被正确归类 → 归属感

**Acceptance Criteria**:
- **AC1**：注册页显式提供两选项——"我是机构/出版社质检岗" / "我是研究生/独立作者"，单选必填
- **AC2**：注册成功后，`用户.角色` 字段落库为 `B-机构质检` 或 `C-个人`
- **AC3**：B 端注册额外要求填写机构名称，不强制认证但留字段以便后期对账
- **AC4**：邮箱必须验证激活，未激活账户不得发起任务
- **AC5**：注册界面**不出现**"自动校对/取代人工"话术（notes #5 产品语言硬约束）

**User-Perceivable Changes**:
- **Before**：陌生的登录页，不知道这工具为谁设计
- **During**：勾选角色类型，感到"这工具理解我是哪一类用户"
- **After**：进入工作台，界面已按角色定制（B 端首屏即有"机构协议"区；C 端首屏是"开始新任务"）

**Constraints**:
- real.md #3（保密）—— 注册即生成用户协议草稿链接，明示数据流向
- notes #5 —— 话术审查

**Dependencies**:
- Requires：无（基座）
- Enables：所有后续 CS

#### MS-L-02：编辑登录并进入工作台

**CS**：CS-00 | **Type**：Light | **Priority**：P1 | **Story Points**：1 | **MVP 复用度**：改造

**User Story**:
作为一个 **已注册编辑**，
我想 **快速登录并看到我的书稿列表**，
以便 **立即恢复上次的工作状态**。

**Light Story 评估**:
- **起始状态**：账户在手但任务散落各处
- **转折点**：登录后直接看到"进行中 / 已完成 / 历史"三栏
- **终止状态**：知道"上次做到哪了"
- **情感弧**：担忧（东西丢没丢）→ 确认仍在 → 掌控

**Acceptance Criteria**:
- **AC1**：登录成功后，工作台首屏按任务状态分三栏——"进行中 / 待确认 / 已完成历史"
- **AC2**：工作台顶部显示角色徽章（B 端显示机构名；C 端显示个人账号）
- **AC3**：登录 session 有效期 B 端较短（默认 4 小时，强制重新鉴权）、C 端较长（7 天），符合 real.md #3 保密强度分级
- **AC4**：登录失败累计 5 次锁定 15 分钟

**User-Perceivable Changes**:
- **Before**：登录框
- **During**：填邮箱密码
- **After**：工作台按角色定制，历史任务可见

**Dependencies**:
- Requires：MS-L-01
- Enables：CS-01 全部

#### MS-D-01：登录失败 / 会话过期 / 账户锁定恢复

**CS**：CS-00 | **Type**：Dark | **Priority**：P1 | **Story Points**：2 | **MVP 复用度**：全新

**User Story**:
作为一个 **遇到登录问题的编辑**，
我想 **看到清晰的失败原因与恢复路径**，
以便 **不会因技术障碍而放弃使用**。

**Dark Story 评估**:
- **起始状态**：想登录继续工作
- **转折点**：密码错误 / 会话过期 / 账户被风控锁定
- **终止状态**：看到明确状态 + 重置/解锁路径
- **情感弧**：期待 → 受阻 → 被引导 → 恢复

**Acceptance Criteria**:
- **AC1**：密码错误提示"邮箱或密码错误"（不区分，防撞库）
- **AC2**：会话过期时跳出登录页并保留**目标页面 URL**，登录后自动回跳
- **AC3**：5 次失败锁定后，邮件告知锁定 + 解锁链接，有效期 1 小时
- **AC4**：**锁定期间不得**静默吞掉请求，API 须返回 `423 Locked` 明码错误

**Dependencies**:
- Requires：MS-L-02
- Blocks：若此故事未实现，任何运维故障会导致编辑直接流失

### CS-01：核校任务全流程（主价值流，P0）

**描述**：编辑把一本书稿从上传到收到三维度独立报告的完整闭环。这是**整个产品的核心价值流**，也是本次迭代最高优先级。

**Core Value**：把"逐行人工翻原典核校"的机械劳动转为"上传 → 按故事推进 → 收到可审计报告"的结构化行动链。

#### MS-L-03：上传书稿文件（多格式）

**CS**：CS-01 | **Type**：Light | **Priority**：P0 | **Story Points**：3 | **MVP 复用度**：已有

**User Story**:
作为一个 **带着书稿来校对的编辑**，
我想 **拖入或选择一个 DOCX/PDF/MD/TXT/EPUB 文件**，
以便 **让系统开始理解我的书稿**。

**Light Story 评估**:
- **起始状态**：手头是散稿，需要逐条人工翻书
- **转折点**：拖进去，系统读懂了
- **终止状态**：系统显示已解析段落数、疑似引文数
- **情感弧**：期待 → 证实系统理解 → 信心建立

**Acceptance Criteria**:
- **AC1**：支持拖拽 + 点击选择，UI 显示当前支持的 5 种格式（`.docx / .pdf / .md / .txt / .epub`）
- **AC2**：上传后 3 秒内显示"正在解析…"，30 秒内给出解析结果卡片（段落数 / 疑似引文数 / 文件大小 / 页数）
- **AC3**：非白名单格式直接拦截并提示，不进入解析管线
- **AC4**：上传过程中可取消，未完成上传的文件立即从服务器删除（对齐 real.md #3）
- **AC5**：超过单文件上限（默认 20MB）给出明确错误，**不得**静默截断

**User-Perceivable Changes**:
- **Before**：空的上传区，手上有散稿
- **During**：拖入，进度条滚动
- **After**：看到卡片"书稿《XX》已解析 / 287 段 / 找到疑似引文 63 条"

**Constraints**:
- notes #3 —— PDF 竖排需检测方向；CJK Ext B-G 字符不丢
- real.md #3 —— 任务取消 = 立即删除

**Dependencies**:
- Requires：MS-L-02
- Enables：MS-L-04

#### MS-L-04：关联参考文献（单份或多份权威参考）

**CS**：CS-01 | **Type**：Light | **Priority**：P0 | **Story Points**：5 | **MVP 复用度**：改造

**User Story**:
作为一个 **以权威参考为绳尺的编辑**，
我想 **上传我选定的参考文献（一份或多份），并标注每份的版本角色**，
以便 **系统以这些参考为唯一判断标准**。

**Light Story 评估**:
- **起始状态**：书稿有一大堆引用，但参考文献散在编辑自己的硬盘/工具书架上
- **转折点**：上传并标注"《论语》-朱熹集注" / "《论语》-杨伯峻译注"等角色
- **终止状态**：参考清单成为本任务的"真值绳尺"
- **情感弧**：散乱 → 有序 → 确信系统用对了底本

**Acceptance Criteria**:
- **AC1**：支持同一任务关联多份参考；每份必填"**版本角色**"下拉——`原典 / 注本 / 现代译本 / 工具书 / 其他`（对齐 cog.md 参考文献分类）
- **AC2**：每份参考附"适用范围"自由文本（如"第 3 章引《论语》用朱熹本"），非必填但可见
- **AC3**：上传后每份参考卡片显式标注"数据流向：本地 / 发送给 DeepSeek（硅基流动）"，对齐 real.md #3
- **AC4**：系统**不**预置任何受版权保护的现代校点本（real.md #5），若用户上传受版权参考，UI 显示"您已声明拥有使用权"复选框
- **AC5**：支持 v1.0 仅走"用户上传"分支；但**后端接口抽象为 `SourceCorpusProvider`**，预留 v1.1 接入外部语料库（ctext.org / 殆知阁）的扩展点

**Perceivable Changes**:
- **Before**：上传了书稿但系统不知"对什么标准"
- **During**：添加 1-N 份参考卡片，每份选角色
- **After**：任务配置完成，知道系统将用这些参考做判断

**Constraints**:
- real.md #4 —— 以上传参考为准绳，系统不越权判"错"，只报"符合 / 不符合 / 未找到"
- real.md #5 —— 版权责任在用户
- notes #3 —— 参考文献同样需要文史字符工程（繁简、异体、Ext B-G）

**Dependencies**:
- Requires：MS-L-03
- Enables：MS-L-05, MS-L-06

#### MS-L-05：预估费用 + 二次确认 + 发起任务

**CS**：CS-01 | **Type**：Light | **Priority**：P0 | **Story Points**：3 | **MVP 复用度**：全新

**User Story**:
作为一个 **按量付费的编辑**，
我想 **在发起任务前先看到清晰的费用预估**，
以便 **对成本有可控预期，不怕账单刺客**。

**Light Story 评估**:
- **起始状态**：上传了书稿和参考，但不知要花多少钱
- **转折点**：系统给出预估 `¥X.XX ± Y%`，附拆分（模型单价 × 引文数估算 × 调用次数）
- **终止状态**：确认后任务启动
- **情感弧**：不确定 → 透明 → 放心确认

**Acceptance Criteria**:
- **AC1**：点击"发起校对"后弹预估卡片，包含：预估费用 / 预估时长 / 引文条数估算 / 拆分公式
- **AC2**：预估 < ¥50 直接确认即可；≥ ¥50 强制二次确认（real.md #6）
- **AC3**：预估卡片底部显示"实际费用将在运行中 SSE 推送；超预估 1.5× 自动暂停"（引到 MS-D-04）
- **AC4**：确认后生成 `校对任务` 记录落库（cog.md 实体），含 `task_id` + 版本戳三元组占位（MAS-5 填充）
- **AC5**：取消后**不**产生费用，未调用 API

**Perceivable Changes**:
- **Before**：不知成本
- **During**：看到拆分明细
- **After**：已启动，进入 SSE 观察态

**Constraints**:
- real.md #6 —— 成本上限与二次确认
- notes #4 —— 任务状态必须持久化（不能只在内存）

**Dependencies**:
- Requires：MS-L-04
- Enables：MS-L-06

#### MS-L-06：SSE 实时观察进度（三段式里程碑）

**CS**：CS-01 | **Type**：Light | **Priority**：P0 | **Story Points**：3 | **MVP 复用度**：已有

**User Story**:
作为一个 **发起了任务的编辑**，
我想 **实时看到校对进度和费用累计**，
以便 **了解任务运行状态、决定是否等待或切走**。

**Light Story 评估**:
- **起始状态**：任务已发起，但在服务端是黑盒
- **转折点**：SSE 推送"解析完成 → 引文提取完成 → 正在校对 24/63..."
- **终止状态**：任务完成，可进入报告
- **情感弧**：不确定 → 可见 → 完成

**Acceptance Criteria**:
- **AC1**：SSE 流有**三段式里程碑**事件：`parse_done` / `extract_done` / `verify_progress`（含 current/total/cost_accumulated）
- **AC2**：每 5 秒最多推一次（防浏览器过载）；没有进展时推心跳
- **AC3**：页面刷新后 SSE **自动重连**，不丢进度（基于 `task_id` 幂等）
- **AC4**：累计费用随 verify 推送增长，与预估并排显示
- **AC5**：任务详情页可随时离开，后端仍运行；回页面继续接 SSE

**Perceivable Changes**:
- **Before**：只知任务在跑
- **During**：进度条 + 当前步骤 + 已用费用
- **After**：任务完成指示出现"查看报告"按钮

**Constraints**:
- notes #4 —— 持久化，浏览器关闭后端继续

**Dependencies**:
- Requires：MS-L-05
- Enables：MS-L-07

#### MS-L-07：三维度独立报告呈现（含参考匹配三态）

#### MS-D-05：客观置信度计算（不依赖 AI 自评）

**CS**：CS-02 | **Type**：Dark | **Priority**：P0 | **Story Points**：5 | **MVP 复用度**：全新

**User Story**:
作为一个 **要用置信度决定复核顺序的编辑**，
我想 **置信度来自可验证的客观指标，而非 AI 自己打的分**，
以便 **不被 AI 自负误导**。

**Dark Story 评估**:
- **起始状态**：MVP 直接 `raw.get("confidence")` 从 AI 响应读——AI 可能虚高
- **转折点**：v1.0 改为多信号融合——`参考命中度 + 跨模型一致性 + 原文定位覆盖度`
- **终止状态**：置信度成为真正的分流利器
- **情感弧**：怀疑（AI 说 95% 真的吗）→ 验证路径清晰 → 敢用

- **AC1**：**删除**从 AI 响应 `confidence` 字段直接读取的逻辑（real.md #2 硬约束）
- **AC2**：置信度由三项加权融合，各项可追溯：

- **AC4**：置信度算法版本号 `confidence_algo_version` 落入任务版本戳（MAS-5）
- **AC5**：低置信度（<0.6）条目自动进入 MS-L-07 的黄色提示带

**Constraints**:
- real.md #2 —— 置信度不得 AI 自评

**Dependencies**:
- Requires：MS-L-07

## 6. 测试指南（Testing Guide）

### 6.1 分类测试策略

| 测试类型 | 覆盖 MS 类型 | 工具 |
|---------|-------------|------|
| 单元测试 | 所有 MS 的核心算法（置信度计算 / 模型拒绝检测 / 幂等 key 生成 / 版本 hash） | pytest |
| 集成测试 | Light 故事的 happy path（上传→发起→SSE→报告） | pytest-asyncio + aiohttp mock |
| 端到端测试 | CS-01 完整旅程 + CS-02 各 Dark 故事触发 | playwright |
| 模糊/压力测试 | 长任务（10 万字+）+ 限流 + 服务崩溃恢复 | 自定义脚本 + locust |
| 安全测试 | 日志脱敏 + PII 泄漏扫描（notes #2）+ 协议合规 | 手工审计 + bandit |

### 6.2 Dark 故事专项测试（本项目高权重）

每个 Dark MS 须有**负面测试用例**：

| MS | 负面测试 |
|----|---------|
| MS-D-01 登录异常 | 5 次错密码后请求必须返回 423 |
| MS-D-02 审核拒绝 | mock DeepSeek 返回审核拒绝 JSON → 报告状态必须为 `REJECTED_BY_MODERATION` |
| MS-D-03 API 重试 | mock 429 限流 → 必须按 15/30/45/60 秒退避 |
| MS-D-04 超额暂停 | 实际费用超预估 1.5× → 任务状态必须切到 `paused` |
| MS-D-05 置信度客观化 | AI 返回 `confidence=0.99`、参考命中度 0.2 → 最终置信度应接近 0.2，证明 AI 自评被丢弃 |
| MS-D-06 主动销毁 | 触发销毁后查询上传文件 → 必须 404 |
| MS-D-07 导出失败 | 触发分片 → 必须生成多文件 zip |

| 术语 | 定义 |
|------|-----|
| **CS（Complex Story）** | 复合故事，3-8 个 MS 组成的完整用户旅程 |

| **Grey（灰色故事）** | 循环日常操作——效率优化主线 |

| **三维度** | 字词准确性 / 解释一致性 / 上下文相符性——本项目校对结果的必拆字段 |
| **参考匹配三态** | MATCH / NOT_MATCH / NOT_FOUND_IN_REF——以上传参考为准绳的中性状态 |
| **版本戳三元组** | model_id + prompt_hash + source_refs_hash——报告不可篡改的身份证 |
| **MVP 复用度** | 已有（原封移植）/ 改造（逻辑保留但需重构）/ 全新（MVP 未实现） |
