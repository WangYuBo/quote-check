---

## 1️⃣ 核心价值主张

现在写的是：**尊重事实 / 人机协作 / [待填]**

**盲区**：
- "尊重事实"太抽象，每个工具都敢这么写。文史校对的真正痛点不在"字词对错"，而在**"版本异文"**和**"语境误用"**——同一句话《论语》朱熹本和皇侃疏本可能就不一样，你不明确立场就会自相矛盾。
- "人机协作"是过程描述，不是价值。**编辑真正怕的不是漏，而是不敢信 AI 的判断**。

**建议补足的第三条**（选一）：
- **"可审计的证据链"**：每个提示都附原典截图/页码/版本号，编辑能三秒内复核。这是唯一能打消资深编辑疑虑的东西。
- **"保留编辑终裁权"**：AI 只标注 + 给置信度，不改稿。这对出版社采购方是合规卖点。
- **"覆盖异文与语境，不止字词"**：定位比单纯 OCR 校对高一档。

我倾向 **"证据链"**——它同时解决信任和合规。

---

## 2️⃣ 目标用户

现在写的是：**文史类图书编辑 / 其他校对人员**

**盲区**：
- "其他校对人员"是定位模糊的信号——**谁付费？** 编辑个人几乎不自费买工具；出版社 ERP 采购周期长、需发票和等保。这两类商业化路径完全不同。
- 漏掉了**付费意愿可能更强**的两类小众用户：
  - **人文社科博士生/青椒**：论文引文错一处就被答辩/审稿人抓把柄，愿意个人付 99-299/年。
  - **自出版学者**（豆瓣阅读、得到、公号专栏作者）：没有编辑团队兜底，最需要 AI 替代校对。
- 漏掉了**真正的 B 端买家**：**高校出版社的质检岗**、**古籍整理项目组**（他们有国家出版基金要结项，漏校会被扣分）。

**建议重写**：
- **核心付费用户**：高校/文史类出版社质检岗（B 端）
- **首发种子用户**：人文社科研究生 + 自出版学者（C 端，冷启动友好）
- **辅助场景**：出版社编辑日常校对（通过 B 端渗透）

先做 C 端 PLG 拉口碑，再切 B 端采购——这比直接卖出版社好走。

---

## 3️⃣ 技术栈

现在写的是：**Next.js 15 + Tailwind + Drizzle + Neon + Vercel AI SDK + Claude/GPT-4**

**盲区（这一条最关键，前面两条都不如这个）**：

1. **真正的技术难点不在前端/DB，而在三件事你没写**：
   - **原典语料库从哪来**？如果没有可索引的原典库，所有"核校"都是海市蜃楼。公版（《四库全书》电子版、ctext.org、殆知阁）+ OCR 质检 + 版本标注，这事本身就是半个项目。
   - **异文处理**：《论语》有皇侃、邢昺、朱熹、刘宝楠四个主流版本，校对结果要绑版本，否则对编辑没用。
   - **长书稿处理**：一本书 30-50 万字，Vercel serverless 函数 **60 秒超时**，直接崩。这个架构跑不起核心流程。

2. **Next.js + Vercel 架构的隐患**：
   - 长任务必须走 **worker 队列**：Inngest / Trigger.dev / 自建 Redis Queue，不能在 route handler 里直接跑。
   - 大文件（docx 20MB+）要 **R2/S3 直传**，别走 Next.js server。
   - PostgreSQL 不够，需要 **pgvector**（语义匹配相似引文）+ **全文检索**（典籍名、章节快速定位）。Postgres 能兜住，但要提前规划索引。

3. **模型选择**（实战经验）：
   - Claude 对古文/文言理解明显优于 GPT-4，但结构化输出稳定性略差。
   - 建议 **Claude 做语义判断 + GPT-4 做结构化校验 + 分歧时人工介入**，而不是二选一。
   - 置信度不能让模型自评（会撒谎），要用**跨模型一致性**或**检索命中度**计算。

**最诚恳的建议**：**先别急着定技术栈，先花 2 周验证原典语料库从哪来、OCR 质量够不够**。如果语料库搞不定，再好的架构也没用；如果能搞定，技术栈反而是最简单的部分。

## 🔴 严重问题

### 1. 关系模型有两处错了

**书稿-引文：多对多** ❌ 
> 原文："某个引文可能存在于多个书稿"

这是把"引文"和"原典中的句子"概念混了。在这个系统里：
- **引文 = 书稿中的一次引用行为**（某人在某本书稿某段落里引了"道可道非常道"）
- 不同书稿即使都引"道可道非常道"，那是**两条不同的引文**（上下文、解释、位置都不同，校对结果也不同）

正确应是 **一对多**：一本书稿有多条引文；每条引文属于且只属于一本书稿。

**引文-参考文献：多对一** ❌
> 原文："某个引文只能与某个参考文献为标准核校"

这个武断了。实际场景：
- "学而时习之"既在《论语》也被《史记》《礼记》转引——一条引文可能有多个候选源
- 版本异文（朱熹本 vs 皇侃本）校对时需要同时比对多个参考文献
- MVP `origin/` 已经支持"多原文管理（最多10部）"就是为了这个

正确应是 **多对多**。

---

### 2. 漏掉了 MVP 已实现的核心实体

MVP 的产出物就是"三维度校对结果 + 置信度 + 修改建议"，但 cog 里完全没出现这个实体。缺失：

- **校对任务 (Task)**：一次完整的校对作业（对应 MVP 里的 task_id、SSE 进度流、导出报告）
- **校对结果 (Verification)**：单条引文的校对输出，包含三维度评分（文字准确性/解释一致性/上下文相符性）+ 置信度 + 改进建议
- **引文上下文段落 (Context Paragraph)**：引文所在的书稿原段落——这是判断"上下文相符性"的必要输入，不是冗余信息

没有这几个实体，整个认知模型就只是一个"数据上传清单"，无法描述"这个系统在做什么"。

---

### 3. 分类全是占位符

```

</书稿>
```

这违反了文件顶部自己写的原则："**分类方式（人类明确，避免AI乱分类）**"。应该是：

- **书稿**：文史通俗读物 / 学术专著 / 古籍整理 / 学位论文 / 公号专栏
- **引文**：直接引用（带引号）/ 间接引用（化用）/ 标注引用（带脚注）——这三类在 MVP `extract_quotes_prompt.txt` 里已经定义
- **参考文献**：原典 / 注本 / 现代译本（版本敏感度依次降低）

---

### 4. 用户分类太浅

现在是"普通用户 / 系统管理员"——这是任何系统都能套的废话。对照 meta.md 里已经分好的用户画像应该是：

### 5. 唯一编码全是 UUID，浪费了业务含义

"唯一编码：UUID" 对 AI 理解业务没任何帮助。更有信息量的：

- **书稿**：`task_id`（上传时生成，对应 MVP 的任务系统）
- **引文**：`{task_id}-{顺序号}`（如 `20260417-001-042`，便于人工复核定位）
- **参考文献**：文件哈希 + 书名 slug（如 `lunyu-yangbojun-zhu`）
- **校对结果**：`{引文id}-{参考文献id}`

**AI 为什么想不到**：大模型开发者很少在产品逻辑里处理"被自己这层审核拦截"的情况——这在国产 AI 应用里是真实发生的事故源。

### 2. 日志不得包含书稿内容——调试路径的隐性泄露
**为什么关键**：工程师 debug 时习惯打 `logger.info(request_body)`。书稿原文进了日志 = 日志存档=泄露面扩大 N 倍。一次 ELK 日志泄露，相当于一次全量用户数据泄露。
**AI 为什么想不到**：AI 写代码默认鼓励"打日志便于调试"；几乎所有 AI 生成的 FastAPI 示例都在打整个请求体。

### 3. 繁简混排、异体字、竖排——文史场景特有的字符工程陷阱
**为什么关键**：
- 书稿简体 vs 原典繁体（"乾坤"简化为"干坤"是经典错误）

**AI 为什么想不到**：训练数据里现代简体中文占压倒性比例，AI 对文史领域的字符工程没有反射弧。
