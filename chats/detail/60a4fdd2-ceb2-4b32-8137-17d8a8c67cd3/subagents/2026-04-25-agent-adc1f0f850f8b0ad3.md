## 特定答案：中文匹配（如 `人法地`）

匹配是**基于 trigram 相似度**，后面跟着**LLM 验证**。它是这样工作的：

1.  **归一化**：引用 `人法地` 在被用作 pg_trgm SQL 查询之前会经过 `normalizeForCompare(text, 'simplified')` + `stripForTrigram()`
2.  **Trigram 相似度**：PostgreSQL `pg_trgm` 扩展将字符串分解为 3 字符的 n-gram（trigram），并计算引用文本和每个参考段落归一化文本之间的 Dice 系数相似度。对于像 `人法地` 这样的中文文本，这是纯字符级别的 n-gram 重叠——**不是语义的，也不是精确的子字符串**，而是来自共享 trigram 序列的统计重叠。`reference_paragraph.text_normalized` 列上的 GIN 索引（`idx_ref_para_text_trgm`，在 `/Users/wangyubo/github-resp/quote-check/lib/db/migrations/_hand_triggers.sql` 第 166 行创建）驱动了这一点。
3.  **基于阈值分类**：
   - `similarity >= 0.75` → 预匹配 = `'MATCH'`
   - `0.40 <= similarity < 0.75` → 预匹配 = `'PARTIAL_MATCH'`
   - `0.30 < similarity < 0.40` → 仍会检索，但预匹配 = `'NOT_MATCH'`
   - `similarity <= 0.30` → 未检索到段落，预匹配 = `'NOT_FOUND_IN_REF'`
4.  **LLM 确认**：无论 trigram 结果如何，LLM prompt 都会收到检索到的段落和引用文本。LLM 进行逐字的**语义评估**，包括处理异文（不同的权威版本）。但是，如果因为上传了参考文档而计算出了 `preMatchStatus`，它会**覆盖** LLM 的 `text_accuracy.match_status`（第 406 行）。

因此，`人法地` 在参考文本中是否匹配取决于参考中是否存在包含足够 trigram 重叠的段落。例如，如果参考包含 `人法地，地法天，天法道，道法自然`，trigram 重叠会很高（~1.0）。如果参考只包含部分内容如 `人法地`，它仍然可以匹配。如果参考包含非常不同的文本，重叠会很低，匹配会失败。

这种方法是混合的：**pg_trgm 提供向量化的近似子字符串/模糊检索**（可扩展至数千个段落），而 **LLM 通过核对版本变体或同义表述来处理细节语义判断**。阈值（0.4 部分，0.75 完全）基于该项目的 CJK 经验测试。
