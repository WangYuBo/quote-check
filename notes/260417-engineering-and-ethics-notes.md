# 工程约束与伦理边界（real.md 的补充）

> 本文件是 `.42cog/real/real.md` 的配套参考清单——**非硬约束级别，但工程/产品团队需参考**。
>
> real.md 收录的是"不守就在现实世界出大事"的高层硬约束（必选 4 + 可选 3，共 7 条，已满）；
> 本文件收录的是**实现层 + 市场层**的补充约束，更偏工程细节或策略，供开发和 PM 对照。
>
> 每条结构：**约束 → 原因 → AI 为什么容易忽略**

---

## 一、工程约束

### 1. 国产 AI 的内容审核边界——被拒时不得伪造结果

**约束**：DeepSeek / 通义 / 文心等国产模型对近代史、政治敏感话题、某些宗教内容会内置审核拒绝。系统必须捕获模型拒绝响应，明确标记该条引文为"内容审核拒绝，无法校对"状态，不得让 AI 返回空结果让用户误以为"AI 没找到问题"。

**原因**：文史书稿涉及晚清-民国史、文革史、宗教经典讨论时，**模型拒绝是真实会发生的事故**。若系统把"拒绝"伪装成"通过"，用户基于错误结论发表后果自负。

**AI 为什么容易忽略**：大模型开发者很少在产品逻辑里处理"被自己这层审核拦截"的情况——国产 AI 应用里的隐性事故源。

---

### 2. 日志不得包含书稿与参考文献原文

**约束**：
- 调试日志禁止打印 request body 中的书稿原文片段
- 结构化日志只记录 task_id、引文 hash、调用结果 metadata，不记录引文文本本身
- 错误日志中的原文引用须截断或脱敏

**原因**：工程师 debug 时习惯 `logger.info(request_body)`。书稿原文进了日志 → 日志进了 ELK / 云平台 / 备份 → **日志泄露 = 全量用户数据泄露**，规模远大于直接攻击数据库。

**AI 为什么容易忽略**：AI 生成的 FastAPI / Flask 示例代码默认会把整个请求体打进日志"便于调试"；几乎所有教程都这么写。

---

### 3. 文史字符工程细节

**约束**：
- **繁简转换**：必须使用 OpenCC 或等价权威转换表，**禁止**使用 `str.translate` 或简单字典。避免"乾坤"被转成"干坤"、"發"被转成"发"后又被转回"髪"这类经典错误。
- **字符集**：必须支持 Unicode CJK Unified Ideographs Extension B-G（很多生僻古字在基本平面之外）；对 Unicode 未收录字符预留 PUA 或图片兜底方案。
- **排版方向**：古籍 PDF 常为竖排，OCR 后文字顺序会错乱。解析前必须检测排版方向。
- **合文与异体字**：识别"卅"（三十）、"廿"（二十）、"囍"等合文，不当作打字错误处理；处理异体字（"為/為/爲"）时用映射表，不直接比对。

**原因**：文史场景的"字"和现代文本的"字"是不同问题。用默认文本处理管线会系统性错判。

**AI 为什么容易忽略**：训练数据里现代简体中文占压倒性比例，AI 对古籍文字工程没有反射弧——写出来的代码默认"UTF-8 就够了"。

---

### 4. 长任务的不可丢失 + API 调用幂等

**约束**：
- **状态持久化**：校对任务状态（已提取引文数 / 已完成校对数 / 失败列表）必须持久化到数据库或持久化队列，而非仅在内存中。
- **断点续跑**：服务重启后能从上次失败点继续，不重新从零开始。
- **API 调用幂等**：每次对硅基流动的调用须携带幂等 key（如 `{task_id}-{quote_id}-{attempt_n}`），重试时后端扣费只生效一次。

**原因**：一次 10 万字校对 = 10-30 分钟 + ¥16 API 成本。服务器崩溃 / 限流重试 → 任务丢失 → **用户花了钱没拿到结果**，这是对用户信任的一次性打击。幂等缺失则会在重试风暴中产生数倍扣费。

**AI 为什么容易忽略**：AI 默认写"同步调用→返回"的流程，不会主动设计崩溃恢复和幂等性；重试逻辑通常只有"延迟+重发"，没有幂等 key。

## 二、伦理与话语权

### 5. 产品口径必须是"辅助"，不得用"替代/自动"

**约束**：
- 对外宣传（官网、公众号、营销素材）**禁用**"自动校对"、"取代人工"、"解放编辑"、"AI 校对机器人"等话术
- 必须使用"辅助校对"、"校对提效工具"、"引用核查助手"等措辞
- 产品界面上，AI 输出的每一条结论旁必须有"AI 建议，请人工复核"的显式提示

**原因**：文史类图书编辑是**话语权极强、认同感极高**的专业群体。一旦宣传口径踩到"取代人工"红线，会立即在编辑圈（豆瓣、小红书、朋友圈长文）引发反扑。文史圈对话语尊严极其敏感——这是市场层硬约束。

**AI 为什么容易忽略**：技术创业者和 AI 本身都倾向于强调"自动化"卖点；但专业工具市场"尊重话语权"比"炫酷卖点"更重要。

---

### 6. 禁用"总分"——三维度评分不得简化为单一数值

**约束**：
- 产品界面禁止展示"综合评分"、"总分"、"通过率"等单一数值
- 三维度（字词 / 解释 / 上下文）必须分别展示
- 导出报告的表格中三维度各为独立列，不得计算加权平均

**原因**：三个维度不是一个量纲的东西——字词错误是事实问题，解释一致性是诠释问题，上下文相符是语境问题。**简化成"80 分"会误导编辑略过重要问题**（比如字词 95 分但上下文 40 分，总分 78，看起来还行，实际是严重断章取义）。专业工具的价值就是展示不同维度的差异。

**AI 为什么容易忽略**：AI 和大众产品直觉都倾向于"综合评分更简洁"，但专业工具反向——**不简化才是价值**。

## 三、可复现性补充

### 7. 校对结果历史快照不可变

**约束**（补充 real.md 第 7 条"版本锁定"的实现细则）：
- 每份校对报告一旦生成，其内容、模型版本、prompt 版本、生成时间戳即被**冻结**存储
- 模型升级或 prompt 迭代后，旧报告在界面上保持原样显示，标注"基于 {模型版本} / {prompt 版本} 生成"
- 不得在后端任务里批量"重跑历史任务以获得更好结果"——若用户需要，必须显式发起新任务，生成新报告，旧报告并存
- 导出的 Word / CSV 报告中必须包含模型版本号 + prompt 版本号

**原因**：real.md 规定了"版本锁定"，本条补充"快照不可变"的实现约束。没有这条，"版本锁定"只是元数据，没有强制力。学术场景下编辑/作者引用校对结论时（"经 AI 核校后三处改动"），必须能回到当时的精确状态。

**AI 为什么容易忽略**：AI 默认"越新越好"，会倾向于"自动帮用户用新模型重跑旧任务"；但学术工具**稳定性 > 先进性**，"替用户做好事"反而是破坏。
